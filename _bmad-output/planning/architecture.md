# Architecture Decision Document

_This document builds collaboratively through step-by-step discovery. Sections are appended as we work through each architectural decision together._

## Project Context Analysis

### Requirements Overview

**Functional Requirements:**
27 FRs across 6 categories. Profile Management (FR1-FR5) and Section Management (FR6-FR10) form the data layer. Template Management (FR11-FR14) operates at startup and on selection. CV Compilation — Preview (FR15-FR18) and Build (FR19-FR22) are the core action paths. Error Handling & Security (FR23-FR25) and Workspace (FR26-FR27) are cross-cutting. The section toggle (FR10) and clone (FR5) are non-trivial data operations requiring careful implementation.

**Non-Functional Requirements:**
- Performance: 5s preview, 10s build, 500ms UI interactions, async compilation (non-blocking)
- Security: Sanitization is a hard architectural invariant — not optional validation. LaTeX shell execution (`\write18`) must be blocked. Localhost-only in v1.
- Reliability: Tiger Style throughout — assertions at every boundary, fail fast, no silent data loss, graceful failure after any compilation error.

**Scale & Complexity:**
- Primary domain: Full-stack web (Python/FastAPI + HTMX MPA + LaTeX subprocess)
- Complexity level: Low-Medium — single user, no auth, no concurrent users
- Estimated architectural components: ~8 (API layer, DB adapter, storage adapter, LaTeX compiler adapter, sanitization pipeline, template discovery, workspace HTMX partials, PDF streaming)

### Technical Constraints & Dependencies

- Docker Compose is the only runtime — TeX Live lives in the container
- All 4 abstraction layer interfaces must be defined before any feature is implemented
- REST API is the sole entry point — no business logic in frontend templates
- OpenAPI spec auto-generated by FastAPI — endpoint design IS the API contract
- Auth activatable via middleware without business logic changes

### Cross-Cutting Concerns Identified

1. **Sanitization pipeline** — mandatory transformation stage wherever user text → LaTeX
2. **Async LaTeX compilation** — subprocess isolation, timeout, stdout/stderr capture
3. **Tiger Style assertions** — boundary assertions in every adapter and pipeline stage
4. **Error propagation chain** — LaTeX errors must surface cleanly from subprocess → adapter → API → HTMX partial
5. **Abstraction layer contracts** — interface definitions must precede all adapters

## Starter Template Evaluation

### Primary Technology Domain

Full-stack web application — Python backend with HTMX MPA frontend. No JavaScript build toolchain. Stack is pre-decided in the PRD; this step documents the scaffolding approach.

### Starter Options Considered

No canonical CLI starter exists for FastAPI + HTMX + SQLite. Manual scaffold is the correct approach. Evaluated and ruled out:
- FastAPI project generators (too opinionated, conflict with HTMX-first design)
- Cookiecutter FastAPI templates (include auth/SPA assumptions incompatible with v1 constraints)

### Selected Approach: Manual Scaffold

**Rationale:** The PRD defines all technology choices explicitly. A manual scaffold gives full control over the 4-layer abstraction pattern and avoids fighting against starter assumptions.

**Initialization:**

```bash
# Repository already initialized — scaffold from existing repo root
docker compose up --build
```

**Architectural Decisions Established:**

**Language & Runtime:** Python 3.12+, async via FastAPI/uvicorn

**Web Framework:** FastAPI — chosen for OpenAPI auto-generation (REST API is the sole entry point per PRD), Pydantic validation, and async-native design.

**ORM & Migrations:** SQLAlchemy 2.x (async) + Alembic — supports the DB abstraction interface; migrations are schema-first.

**Frontend Approach:** Jinja2 templates serving HTMX partials — no JS bundler, no SPA. Templates are thin consumers of the REST API.

**Testing Framework:** pytest + httpx (async test client). Unit tests co-located next to source files using `*.test.py` naming convention (configured via `pyproject.toml`). Integration and API tests in root-level `tests/` folder.

```toml
[tool.pytest.ini_options]
python_files = ["*.test.py"]
```

**Linting & Typing:** ruff (linting + formatting) + mypy (type checking), configured in `pyproject.toml`.

**Dependency Management:** uv or pip + requirements.txt — Docker Compose installs dependencies in container at build time.

## Core Architectural Decisions

### Decision Priority Analysis

**Critical Decisions (Block Implementation):**
- Table-per-type data model with full schema defined before any feature
- All 4 abstraction layer interfaces defined before adapters
- LaTeX async subprocess approach

**Important Decisions (Shape Architecture):**
- Preview PDF delivery mechanism
- Compilation progress feedback strategy
- Error response format standard

**Deferred Decisions (Post-MVP):**
- S3 storage adapter
- Postgres DB adapter
- Cloud LaTeX compiler adapter
- Authentication middleware activation

### Data Architecture

**Model: Table-per-type, 3-layer hierarchy (Profile → Section → Entry)**

No polymorphic tables, no type columns, no JSON fallback. Each section type is an explicit table with its own schema, SQLAlchemy model, Pydantic schema, and service method. Adding a new section type requires a migration + model + schema + service — intentional, not accidental.

```
profiles
  id, name, description, created_at, updated_at

profile_contacts                ← profile-level, always present (CV header)
  id, profile_id,
  type (enum: email | phone | github | linkedin | website | twitter | ...),
  value, order_index

── Structured Sections (each has its own *_sections + *_entries table pair) ──

experience_sections / experience_entries
  entries: title, organisation, start_date, end_date, bullets (JSON), order_index

education_sections / education_entries
  entries: institution, degree, field, start_date, end_date, order_index

project_sections / project_entries
  entries: title, description, tech_stack, url, order_index

skill_sections / skill_entries
  entries: name, proficiency (enum: beginner | intermediate | advanced | expert), order_index

certification_sections / certification_entries
  entries: name, institution, issue_date, expiry_date, credential_id, order_index

language_sections / language_entries
  entries: language, proficiency (enum: native | fluent | conversational | basic), order_index

publication_sections / publication_entries
  entries: title, venue, date, co_authors, url, order_index

── Free Text Sections ──

free_text_sections
  id, profile_id, label (summary | objective | ...),
  content (TEXT), is_enabled, order_index
```

All section tables share: `id, profile_id, is_enabled, order_index`.

**Ordering:** Integer `order_index` — simple, acknowledged trade-off (gaps/conflicts possible, safe for single-user personal tool).

**ORM & Migrations:** SQLAlchemy 2.x (async) + Alembic. Schema-first — migrations generated from model changes.

### Authentication & Security

**v1:** No authentication. FastAPI middleware hook exists in `apis/` but is inactive. No user-session assumptions anywhere in business logic.

**Sanitization (hard invariant):** All user-provided text passes through the sanitization pipeline before reaching any LaTeX template. This is enforced architecturally — the LaTeX compiler adapter interface accepts only pre-sanitized input. `\write18` and `\immediate\write18` are blocked at the sanitization layer.

### API & Communication Patterns

**REST API** — FastAPI auto-generates OpenAPI spec. Endpoint design IS the API contract.

**HTMX response strategy:** `HX-Request` header check lives exclusively in `controllers/`. Controllers call services, then decide response format:
- `HX-Request` present → return Jinja2 HTML partial
- No header → return JSON

**Compilation progress (FR18):** HTMX polling every 500ms against a `/compile/status/{job_id}` endpoint. No SSE, no WebSocket.

**Preview PDF delivery (FR17):** Compiled PDF streamed as bytes from `/preview/{job_id}`. HTMX swaps in `<embed src="/preview/{job_id}">` partial. Browser renders PDF natively.

**LaTeX subprocess:** `asyncio.create_subprocess_exec` — non-blocking, captures stdout/stderr, enforced timeout.

**Error response format (standard across all endpoints):**
```json
{"error": "compilation_failed", "message": "Human-readable summary", "detail": "<full latex log>"}
```
HTMX controllers render this as an error partial. JSON clients receive the raw object.

### Frontend Architecture

HTMX + Jinja2 MPA. No JS bundler, no SPA, no client-side state management. Server is the source of truth. Templates are thin — no business logic, only rendering.

Split-view workspace: left panel (profile editor form), right panel (`<embed>` PDF preview). Both driven by HTMX partials.

### Infrastructure & Deployment

**Runtime:** Docker Compose exclusively. TeX Live in the container. Volume mounts for `assets/templates/` and `assets/outputs/`.

**No CI/CD in v1.** Local development only.

**Logging:** Python `logging` module, structured log output to stdout (captured by Docker).

**Environment config:** All configuration via environment variables, validated at startup by `configs/` using Pydantic Settings. App fails fast on missing required vars.

### Decision Impact Analysis

**Implementation Sequence:**
1. Define all 4 abstraction layer interfaces (`interfaces/`) — blocks everything else
2. Scaffold project structure + Docker Compose + health endpoint
3. DB adapter (SQLite) + all table migrations
4. Profile + contacts CRUD (data foundation)
5. Section CRUD per type (builds on profile)

8. Template auto-discovery + validation

**Cross-Component Dependencies:**
- Sanitization pipeline must be complete before any compilation path
- All section tables must exist before workspace can function
- Interface definitions must precede all concrete adapters

## Implementation Patterns & Consistency Rules

### Naming Patterns

| Context | Convention | Example |
|---|---|---|
| DB tables | `plural_snake_case` | `experience_sections`, `profile_contacts` |
| DB columns | `snake_case` | `profile_id`, `order_index`, `is_enabled` |
| Foreign keys | `{table_singular}_id` | `section_id`, `profile_id` |
| API endpoints | `plural_snake_case` | `/profiles`, `/profiles/{id}/experience-sections` |
| Route params | `{id}` FastAPI style | `{profile_id}`, `{section_id}` |
| Python functions | `snake_case` | `get_profile()`, `create_experience_entry()` |
| Python classes | `PascalCase` | `ExperienceSectionSchema`, `ProfileService` |
| Python files | `snake_case` | `experience_section.py`, `profile.test.py` |
| JSON fields | `snake_case` | `{"profile_id": 1, "is_enabled": true}` |
| Jinja2 templates | `snake_case` | `experience_section.html`, `profile_list.html` |

### Format Patterns

- **Dates:** ISO 8601 strings always — `"2024-01"`, `"2024-01-15"`. Never timestamps.
- **Booleans:** `true/false` — never `1/0`.
- **Success response:** Direct object — no wrapper. `{"id": 1, "name": "..."}`
- **Error response:** `{"error": "error_code", "message": "Human summary", "detail": "..."}` — always this shape.
- **HTTP status codes:** `200` success, `201` created, `422` validation error (FastAPI default), `500` compilation failure, `404` not found. Never `200` with an error body.

### OOP Design Patterns

#### Interfaces — Protocol (structural typing)

Use `typing.Protocol` for all 4 abstraction layer interfaces. No `ABC`, no explicit inheritance required — structural typing like TypeScript interfaces.

```python
# interfaces/database.py
from typing import Protocol
from app.schemas.profile import Profile, CreateProfileSchema

class DatabaseProtocol(Protocol):
    async def get_profile(self, profile_id: int) -> Profile | None: ...
    async def create_profile(self, data: CreateProfileSchema) -> Profile: ...
```

Adapters implement the Protocol by matching the method signatures — no `(DatabaseProtocol)` in the class definition required.

#### Adapters — Standalone classes, no BaseAdapter

Each adapter is a standalone class. No shared `BaseAdapter` parent — no real shared behaviour exists. Shared utilities go in `utils/` as plain functions.

```python
# adapters/sqlite_database.py
class SQLiteDatabaseAdapter:  # no BaseAdapter inheritance
    def __init__(self, url: str):
        self._url = url

    async def get_profile(self, profile_id: int) -> Profile | None:
        ...
```

#### Factories — Plain functions, not factory classes

```python
# adapters/factories.py
def create_db_adapter(config: Settings) -> DatabaseProtocol:
    if config.db_backend == "sqlite":
        return SQLiteDatabaseAdapter(config.sqlite_url)
    assert False, f"unknown db_backend: {config.db_backend}"  # Tiger Style — fail fast
```

#### Services — Standalone classes, constructor injection

No `BaseService`. Each service takes its adapter dependencies via constructor. No direct env reads, no adapter instantiation inside services.

```python
# services/profile_service.py
class ProfileService:
    def __init__(self, db: DatabaseProtocol):
        self._db = db

    async def create_profile(self, data: CreateProfileSchema) -> Profile:
        assert data.name, "profile name must not be empty"
        result = await self._db.create_profile(data)
        assert result.id is not None, "inserted profile has no id — db invariant violated"
        return result
```

#### Dependency Injection — FastAPI Depends()

All wiring lives in `apis/dependencies.py`. Routers declare dependencies, never instantiate them.

```python
# apis/dependencies.py
def get_db_adapter(settings: Settings = Depends(get_settings)) -> DatabaseProtocol:
    return create_db_adapter(settings)

def get_profile_service(db: DatabaseProtocol = Depends(get_db_adapter)) -> ProfileService:
    return ProfileService(db=db)

# apis/profiles.py
@router.get("/profiles/{profile_id}")
async def get_profile(
    profile_id: int,
    service: ProfileService = Depends(get_profile_service)
):
    return await service.get_profile(profile_id)
```

### Tiger Style Process Patterns

#### Assertions vs Error Handling

| | Assertion | Error Handling |
|---|---|---|
| **Purpose** | Programmer bug / violated invariant | Expected external failure |
| **On failure** | Crash (`AssertionError`) | Structured error returned to caller |
| **Keyword** | "This should never happen" | "This can happen" |
| **Python** | `assert condition, "message"` | `raise HTTPException(...)` |

**Rules (from Tiger Style source):**
- Minimum 2 assertions per function — one input precondition, one output postcondition
- Assert both input AND output — validate what comes in AND what goes out
- Pair assertions — assert before write AND after read from DB/disk
- Split compound assertions — one condition per `assert`, never `assert a and b`
- Cover negative space — assert what should NOT be true, not just what should be true
- **Assertions stay enabled in production** — a crash on bad state beats silent corruption

```python
async def compile(self, sanitized_tex: str) -> bytes:
    # Input assertion (precondition)
    assert sanitized_tex, "tex content must not be empty — sanitization must run first"

    result = await asyncio.create_subprocess_exec(...)
    pdf_bytes = await result.stdout.read()

    # Output assertion (postcondition — programmer invariant)
    assert pdf_bytes, "compiler returned empty output with exit code 0 — invariant violated"
    return pdf_bytes

# Error handling — expected path, NOT an assertion
def get_profile(self, profile_id: int) -> Profile | None:
    return self._db.get(profile_id)  # None is valid — controller raises 404
```

#### Loop Safety

Every loop requires a max iteration constant from `constants/limits.py`. Exceeding the bound is an assertion failure.

```python
# constants/limits.py
MAX_TEMPLATE_DISCOVERY_ITERATIONS = 1_000
MAX_BULLET_POINTS_PER_ENTRY = 20
MAX_SECTIONS_PER_PROFILE = 50
MAX_ENTRIES_PER_SECTION = 100
MAX_COMPILE_POLL_ATTEMPTS = 20  # 20 × 500ms = 10s max

# usage
for i, path in enumerate(template_dir.iterdir()):
    assert i < MAX_TEMPLATE_DISCOVERY_ITERATIONS, (
        "template discovery exceeded max iterations — possible filesystem issue"
    )
```

### Enforcement Rules — All Agents MUST

- Never read `os.environ` directly — always import from `configs/`
- Never put business logic in `apis/` or `templates/`
- Never put HTTP/response logic in `services/`
- Never call the LaTeX adapter without passing through sanitization first
- Never swallow exceptions — log and re-raise or convert to structured error
- Never create `BaseService` or `BaseAdapter` — standalone classes only
- Never hardcode loop bounds inline — always use a constant from `constants/limits.py`
- `HX-Request` header check lives exclusively in `controllers/` — never in `apis/` or `services/`

### Architectural Boundaries

**Data flow:**
```

              views/ partial (HTMX) or JSON response
```

**Sanitization boundary:** `utils/sanitizer.py` is called inside `adapters/pdflatex_compiler.py` only. `LatexCompilerProtocol` accepts only pre-sanitized text — enforced by interface contract.

**Configuration boundary:** Only `configs/settings.py` reads env vars. All other modules import `Settings`.

**Response format boundary:** `HX-Request` header check lives only in `controllers/`.

**SQLAlchemy Base boundary:** `models/base.py` is the ONLY definition of `Base`. All models import from it. Never call `declarative_base()` or `DeclarativeBase` anywhere else.

**Section router boundary:** `apis/sections/__init__.py` auto-registers all section routers. `main.py` includes only the single `sections_router` — new section types are automatically discoverable.

**Profile contacts boundary:** Contacts CRUD lives in `apis/profiles.py` and `controllers/profile_controller.py` as sub-resource routes (`/profiles/{id}/contacts`). No separate contacts file.

**Auth middleware boundary:** `app/middleware/auth.py` is an empty stub. Auth is activated by registering this middleware via config — no business logic changes required (TC7).
